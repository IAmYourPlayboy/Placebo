# Placebo — Closed Alpha Design

> **Статус:** черновик, ожидает review от пользователя
> **Дата:** 2026-05-14
> **Автор:** Claude (Opus 4.7) совместно с IAmYourPlayboy
> **Горизонт:** 4–8 недель до играбельной закрытой альфы для 10–50 друзей

## 1. Контекст и цели

Placebo – десктопное приложение для совместного просмотра живых камер мира. Символ – белый голубь (свобода передвижения, ощущение компании когда ты один).

На момент дизайна существует значительный объём уже написанного кода:

- **Фронт:** Tauri 2 + React 18 + TypeScript, 7 экранов-заготовок, 3D-мир с OSM-тайлами (Three.js + R3F), HLS-прокси через yt-dlp в Vite-middleware.
- **Бэк:** axum + sqlx (PostgreSQL+PostGIS) + Redis + deadpool. 7 миграций (cameras, users_auth, rooms, recordings_clips_ratings, seed, auth_password). Handlers и services под auth, cameras, rooms, clips, boosts, ratings, world. Auth-эндпоинты (register/login/logout/refresh/password-reset) реализованы в последних коммитах.
- **Дизайн:** Figma-макеты 9 ключевых экранов (Welcome, Create account, Main, Categories, Movie card, Channel/stream, Profile, Subscription, Create hub).

Альфа **не переписывает всё с нуля** – она добавляет shell-обёртку по Figma и доводит end-to-end-сценарий до работающего состояния.

### Главный сценарий, который должен работать к концу альфы

```
1. Пользователь A открывает Placebo, регистрируется через email+пароль.
2. Оказывается на Welcome → Main. Видит карточки "Открытых комнат" и "Популярное".
3. Идёт в Категории → Онлайн карта мира → проваливается в 3D-мир с камерами.
4. Кликает на маркер камеры в Токио, видит живой HLS-поток в 3D-плоскости.
5. Жмёт "Смотреть вместе" → создаётся комната, копируется ссылка-приглашение.
6. Пользователь B получает ссылку (в Telegram/мессенджере), открывает Placebo, регистрируется.
7. После клика по ссылке попадает в ту же комнату, видит ту же камеру.
8. A переключает камеру → у B мгновенно обновляется.
9. Оба пишут в чате в комнате.
10. Счётчик viewers_now на камере = 2.
11. B добавляет A в друзья по никнейму.
12. Оба выходят, комната закрывается.
```

Если этот сценарий работает без белых экранов и сбрасываний сессии – альфа готова.

### 2.2 Nice-to-have (если успеваем)

- Избранное камер (bookmark): локально в SQLite, позже сервер.

- Picture-in-picture для видео.

- Платежи, Premium-подписка (экран подписки из Figma НЕ делаем).
- OAuth через соцсети (иконки в Welcome показываем как disabled, клик → toast "скоро").
- Гостевой режим ("Попробовать без аккаунта" – disabled).
- Карточка фильма (Violet Evergarden и т.п.) – это не camera-контент, отдельный pipeline.
- Карты слоёв: светозагрязнение, землетрясения, MCC, Wi-Fi.
- Видеорегистратор (dashcam upload).
- Мобильная версия.
- ElevenLabs-перевод речи.
- Multi-region / anti-DDoS (работаем на одной машине с Cloudflare впереди).
- Собственная генерация 3D Tiles в R2 (оставляем OSM-тайлы через PostGIS, как сейчас).
- Кодеки H.265/AV1 в клиенте (для альфы только H.264 HLS).

## 3. Архитектура shell

### 3.1 Концепция

Shell – это **внешняя рамка приложения**, которая окружает любой экран. В Figma shell включает:

- Левый сайдбар (ширина ~224px, фиксированная).
- Верхнюю панель (навигация `< > ↻`, поиск, переключатель темы).
- Ряд табов над контентом.
- Breadcrumbs под табами.
- Контентная область.

Shell живёт на одном уровне со всем приложением и **не размонтируется** при переходах. Меняется только содержимое contentной области (и breadcrumbs под текущий экран).

### 3.2 Модель табов

Табы – это **настоящие браузерные табы (вариант C)**:

- Каждый таб имеет собственный **React Router memory router** (`createMemoryRouter` из react-router-dom).
- Каждый таб хранит своё состояние в React-дереве через контекст таба (`TabContext`).
- Неактивные табы **не размонтируются**, их React-дерево остаётся живым в DOM (через CSS `display: none` или `<div hidden>`).
- **HLS-видео** в неактивном табе продолжает декодить и буферизоваться (решение о звуке – в deferred decisions; по умолчанию звук приглушён в неактивном табе, финальная политика выбирается позже).
- **3D-рендер** в неактивном табе приостанавливается: сцена не рендерится, GPU-нагрузки нет. Состояние сцены (позиция камеры, загруженные тайлы, активная HLS-текстура) сохраняется, при возврате на таб сцена восстанавливается. Это виртуализация WebGL-контекста – подробнее в разделе 5.3.

**Структура данных таба:**

```ts
type Tab = {
  id: string;                      // uuid
  title: string;                   // отображается в таб-баре
  icon?: string;                   // опционально
  router: MemoryRouter;            // персональная история навигации
  createdAt: number;
  isActive: boolean;
  // runtime state: HLS players, 3D scene state, form state
  state: TabState;
};

type TabManager = {
  tabs: Tab[];
  activeTabId: string;
  openTab(initialPath: string, title?: string): Tab;
  closeTab(id: string): void;
  activateTab(id: string): void;
  renameTab(id: string, title: string): void;
};
```

**Взаимодействие с роутером:** вместо одного `<BrowserRouter>` на приложение, мы используем **один `<RouterProvider>` на активный таб**. Переключение таба = замена router-instance.

**История таба (кнопки `< > ↻`):**

- `<` → `router.navigate(-1)` (идёт по memory history stack внутри таба).
- `>` → `router.navigate(1)`.
- `↻` → `router.revalidate()` (перезагружает данные текущего лоадера).

**Создание нового таба:**

- Клик "+" в таб-баре → создаём пустой таб с путём `/home`.
- Клик на карточку фильма/камеры с модификатором Ctrl (или из контекстного меню) → открываем в новом табе.

**Закрытие таба:**

- Крестик на табе → `closeTab(id)`.
- Если закрыт активный таб → активируется соседний справа (или слева, если он крайний правый).
- Если закрыт последний таб → автоматически открывается новый таб `/home`.

### 3.3 Сайдбар

Структура из Figma (сверху вниз):

- **Верхний блок** (контекстные быстрые ссылки, над списком):
  - Уведомления (с badge)
  - Профиль
- **Основной список** (с hover-выделением "пузырьком"):
  - Главная
  - Создать
  - Категории
  - Люди
- **Нижний блок** (после разделителя):
  - История
  - Избранное
  - Папки
- **Подвал:**
  - Настройки
  - Иконка-бургер (в левом нижнем углу, для сворачивания сайдбара – **не в альфе**, но место в layout оставляем).

Клик на любой пункт сайдбара:
- Если **уже открыт таб** с этим разделом → активируется он.
- Если **нет** → открывается новый таб с соответствующим path.

### 3.4 Верхняя панель

- Слева: `< > ↻` – см. раздел 3.2 про историю таба.
- Центр: поисковая строка. В альфе – не работает (plain input + placeholder + disabled submit). Занимает место в layout.
- Справа: переключатель темы. В Figma визуально показан как трёхпозиционный (луна / тумблер / солнце), что соответствует **тёмная / auto / светлая**. В альфе реализуем все три состояния: "auto" следует системной теме ОС.

### 3.5 Темы

**Две темы в альфе:** светлая и тёмная. Третий вариант "auto" = следовать системной теме ОС.

Реализация:
- CSS custom properties (CSS variables) для всех цветов.
- Два набора переменных: `[data-theme="light"]` и `[data-theme="dark"]`.
- `ThemeProvider` (React context) хранит выбор пользователя и синхронизирует его с `<html data-theme="...">`.
- Выбор темы сохраняется в локальном SQLite (в Tauri) или fallback в `localStorage`.
- Режим "auto" слушает `window.matchMedia('(prefers-color-scheme: dark)')`.

**"Тема-кирпич" (игровая тема в духе Warcraft)** – отложено, **не в альфе**. Архитектурно предусматриваем, что тем может быть больше двух (не хардкодим `"light" | "dark"`, используем string-тип с enum-валидацией).

### 3.6 i18n

- Библиотека: `react-i18next` + `i18next`.
- Стартовый язык: русский (`ru`).
- Файлы переводов: `src/i18n/locales/ru.json`.
- Ключи: плоская структура с точками (`home.title`, `sidebar.main`, `auth.login.email`). Не nested – проще для search&replace.
- Никаких строк напрямую в JSX – только через `t('...')`.
- Переключатель языка в Настройках (в альфе можно ограничить одним языком, но инфраструктура готова).

## 4. Архитектура реального времени

### 4.1 WebSocket-канал

**Один WebSocket на сессию пользователя.** После логина клиент открывает соединение к `wss://host/ws` с JWT-токеном в query или заголовке. Соединение мультиплексирует все real-time подписки пользователя: комнаты, viewer-счётчики, уведомления.

### 4.2 Протокол

**Тип сообщений: JSON со structured types, генерируемыми из Rust.**

Все сообщения имеют обёртку:

```rust
// crates/placebo-shared/src/ws.rs
#[derive(Serialize, Deserialize)]
#[serde(tag = "type", content = "payload")]
pub enum ClientMessage {
    // комнаты
    RoomJoin { room_id: Uuid },
    RoomLeave { room_id: Uuid },
    RoomChangeCamera { room_id: Uuid, camera_id: Uuid },
    RoomChatSend { room_id: Uuid, text: String },
    // присутствие
    CameraWatchStart { camera_id: Uuid },
    CameraWatchStop { camera_id: Uuid },
    // ping
    Ping,
}

#[derive(Serialize, Deserialize)]
#[serde(tag = "type", content = "payload")]
pub enum ServerMessage {
    RoomStateSnapshot { room_id: Uuid, state: RoomState },
    RoomCameraChanged { room_id: Uuid, camera_id: Uuid, by_user: Uuid },
    RoomChatMessage { room_id: Uuid, from: Uuid, text: String, at: DateTime<Utc> },
    RoomMemberJoined { room_id: Uuid, user: UserSummary },
    RoomMemberLeft { room_id: Uuid, user_id: Uuid },
    CameraViewersUpdate { camera_id: Uuid, count: u32 },
    Error { code: String, message: String },
    Pong,
}
```

**Генерация TypeScript-типов:**

- **Единый инструмент для всего: `ts-rs`** (crate) – добавляет derive-макрос, который при `cargo test --features export-types` дампит `.ts` файлы.
- `ts-rs` работает и для WebSocket-enum, и для REST request/response структур. Один инструмент = простой build pipeline.
- Путь: `crates/placebo-shared/bindings/` (auto-generated) → копируется/симлинкается в `src/types/api/`.
- Build step в `package.json`: `"gen-types": "cd crates/placebo-shared && cargo test export_bindings --features export-types"`.
- Вызов `gen-types` добавляется в `predev` и `prebuild` хуки npm, чтобы типы всегда свежие.
- `utoipa` оставляем для генерации человекочитаемой OpenAPI-документации (`/api/v1/docs`), но не для генерации клиентских типов.

### 4.3 Серверная часть

- Модуль `crates/placebo-api/src/websocket/` (уже есть).
- На каждый WS-подключение – tokio task, читающий и пишущий.
- Состояние комнат: **Redis pub/sub** (`room:{uuid}` канал). Когда пользователь в комнате меняет камеру → сервер публикует в канал → все задачи, слушающие канал, отправляют сообщение своим WS-клиентам.
- **Presence** (кто в комнате сейчас): Redis `SET room:{uuid}:members` (user_id). TTL 30 секунд, обновляется каждым ping.
- **Viewer counts** (`camera:{id}:viewers`): Redis sorted set по timestamp, периодически чистим expired.

### 4.4 Клиентская часть

- Один хук `useWebSocket()` на всё приложение (в App-root).
- Хук `useRoom(roomId)` подписывается на события комнаты через Context.
- Хук `useCameraViewers(cameraId)` подписывается на viewer-updates.
- При дисконнекте – **exponential backoff reconnect**: 1s, 2s, 4s, 8s, 16s, max 30s.
- При reconnect: клиент отправляет `RoomJoin` для всех активных комнат (идемпотентно).

### 4.5 Синхронизация комнаты

**Модель:** "host wins". Пользователь, создавший комнату – host. Только host может менять камеру. Остальные видят, что выбрал host.

- Для альфы этого достаточно.
- В будущем: любой член комнаты может менять (democratic), или voting, или host может делегировать права.

**Чат:**

- Каждое сообщение в чате идёт через `RoomChatSend` → сервер валидирует → броадкаст через `RoomChatMessage`.
- История чата: **последние 50 сообщений** в Redis (`LIST` с `LPUSH` + `LTRIM`). Загружается при `RoomJoin` как часть `RoomStateSnapshot`.
- Персистентность чата в PostgreSQL – не в альфе.

## 5. Архитектура медиа и 3D

### 5.1 HLS-прокси на бэкенде

Задача: у нас есть публичный HLS-источник камеры (например, YouTube Live или EarthCam). Браузеру нужен URL без CORS-проблем, под нашим доменом.

**Реализация в axum:**

```
GET /api/v1/hls-proxy/{camera_slug}                    → возвращает .m3u8
GET /api/v1/hls-proxy/{camera_slug}/seg/{segment_path} → возвращает .ts
```

- Для каждой камеры в БД хранится `stream_source_type` + `stream_source_config` (JSON): `{type: "youtube_live", video_id: "..."}` или `{type: "direct_hls", url: "..."}`.
- Сервер хранит разрешённый HLS-URL в Redis с TTL 30 минут (обновляем при истечении).
- Сегменты проксируются stream'ом (без буферизации всего сегмента в памяти).
- Для YouTube Live: используем yt-dlp в Rust (через tokio::process::Command) или переписываем логику сами. На старте – yt-dlp через command.

**Важно:** yt-dlp нарушает ToS YouTube, если использовать в production. Для закрытой альфы с друзьями это приемлемо, но **после альфы надо переходить на партнёрские HLS-источники или свой FFmpeg-ingest**. Это зафиксировано в deferred decisions.

### 5.2 Seed-камеры для альфы

Цель: 10–15 реальных + 3–5 тестовых.

Источники для поиска реальных публичных HLS:

- **Skyline Webcams** (skylinewebcams.com) – даёт iframe embed с HLS внутри, можно вытащить.
- **EarthCam** (earthcam.com) – похожий подход.
- **Муниципальные traffic-cams**: у многих городов (например, NYC, Helsinki, Tokyo) есть открытые API. Список собираем в `data/cameras-seed.json`.
- **YouTube Live** (24/7 стримы типа Shibuya, Times Square, Helsinki Senate Square) – через yt-dlp.

Тестовые loop-видео:
- Выложить 3–5 MP4 на сервер (или R2 после настройки), превратить в HLS через FFmpeg батчом, использовать для демо-сценариев где нужен предсказуемый контент.

### 5.3 3D-мир: виртуализация при табах

**Проблема:** WebGL-контексты ограничены (Chromium: 8–16 на приложение). При нескольких табах с 3D-миром мы быстро упрёмся.

**Решение: один общий WebGL-контекст + N логических сцен.**

- Корневой `<Canvas>` монтируется **в shell**, над контентной областью (абсолютно позиционирован, z-index ниже UI).
- Активный таб с 3D-миром "получает" этот canvas через портал/context, рендерит свою сцену.
- Неактивные табы хранят только **state** (позиция камеры, список загруженных тайлов, активная HLS-текстура) в своём TabContext – ничего не рендерят.
- При переключении таба:
  1. Старый таб → сохранить сцену state.
  2. Активный таб → восстановить state в общий canvas.

**Код:** абстракция `useScene3D(sceneId)` – хук, который при mount регистрирует сцену в `Scene3DRegistry`, при unmount снимает. Один `<GlobalCanvas>` рендерит текущую активную сцену.

Это значительная работа, но альтернативы нет.

### 5.4 HLS-плеер в 3D-плоскости

Текущая реализация (`CameraFrustum.tsx`) использует `<video>` + hls.js + `THREE.VideoTexture`. Это работает. Оптимизации (WebCodecs, OffscreenCanvas) – отложены.

Правила для альфы:
- **Один активный HLS-плеер** на таб (на камеру, которую пользователь смотрит).
- Thumbnail-превью других камер в 3D-мире – **статические JPG**, не видео (уже есть).
- При уходе со сцены (переход в другой раздел внутри того же таба) – hls.js destroy, video.pause.
- При переключении на другой таб с 3D-миром: продолжается (п. 3.2, "видео продолжается в фоне").

### 5.5 Отложенные медиа-оптимизации

- **WebCodecs API**: когда понадобится декодить H.265/AV1 или делать frame-perfect синхронизацию в комнатах.
- **OffscreenCanvas + Web Worker**: когда 3D-рендер начнёт тормозить главный поток.
- **SharedArrayBuffer + Atomics**: если понадобится быстрый IPC между табами.
- **Собственный FFmpeg-ingest**: когда появятся первые партнёрские RTSP-камеры.

## 7. Data flow и API

### 7.1 Что хранится локально (Tauri SQLite)

Распространяется на одно устройство пользователя, не синхронизируется:

- `user_preferences`: тема, язык, другие UI-предпочтения.
- `auth_session`: access-token + refresh-token (зашифрованы на уровне OS keychain через Tauri plugin).
- `tabs_state`: список открытых табов на момент закрытия приложения (для восстановления при запуске).
- `recent_searches`: история поиска (на будущее).
- (Отложено: bookmarks, watch_history.)

### 7.2 Что хранится на сервере (PostgreSQL)

Всё общее: users, cameras, rooms, room_members, chat_history (last 50 в Redis, потом в PG), friendships.

### 7.3 Redis

- Сессии (JWT blacklist для logout).
- Viewer-counts.
- Room presence (members online).
- Last chat messages (LRU-кеш 50 сообщений).
- HLS-URL cache (YouTube yt-dlp outputs, TTL 30 min).
- Rate limits.

### 7.4 REST API (уже существующие + нужные)

Существующие handlers: `auth`, `cameras`, `rooms`, `clips`, `boosts`, `ratings`, `users`, `world`, `health`. Часть из них – расширяем, часть – добавляем.

Для альфы нужно (финальный список будет детально расписан в implementation plan):

- `POST /api/v1/auth/register` – есть.
- `POST /api/v1/auth/login` – есть.
- `POST /api/v1/auth/logout` – есть.
- `POST /api/v1/auth/refresh` – есть.
- `GET /api/v1/me` – проверить / добавить, для получения текущего пользователя.
- `GET /api/v1/cameras` – список публичных камер (paginated).
- `GET /api/v1/cameras/:id` – одна камера.
- `GET /api/v1/hls-proxy/:slug` + `/seg/:path` – HLS-прокси.
- `POST /api/v1/rooms` – создать комнату { camera_id }.
- `GET /api/v1/rooms/:id` – информация о комнате.
- `POST /api/v1/rooms/:id/invite` – получить ссылку-приглашение (короткий код).
- `GET /api/v1/rooms/by-invite/:code` – разрешить код в room_id.
- `GET /api/v1/world/tile` – уже есть (OSM-тайлы).
- `GET /api/v1/friends` – список друзей.
- `POST /api/v1/friends` – добавить по никнейму.
- `DELETE /api/v1/friends/:id` – удалить.
- `GET /api/v1/users/:username` – публичный профиль.
- `GET /ws` – WebSocket.

### 7.5 Tauri IPC

Ранее было 8 команд по камерам (рудименты прототипа). Большая часть не нужна – мы переходим на сервер. **Оставить только то, что специфично для десктопа:**

- Работа с локальной SQLite (user_preferences, tabs_state).
- OS keychain для auth-токенов.
- (Потом: уведомления OS, фоновый режим, системный tray, обновления.)

Рудиментарные команды (`greet`, `get_public_rooms`, `create_room`) – удалить в рамках альфы.

## 8. Безопасность и авторизация

### 8.1 Пароли

- Argon2id (уже настроено в бэкенде).
- Минимум 8 символов, без обязательных классов (современный подход: длина > сложности).
- Rate limit на login: 5 попыток за 15 минут на IP + 10 попыток за 15 минут на email (Redis counters).

### 8.2 JWT

- Access-token: 15 минут, подписан HS256.
- Refresh-token: 30 дней, хранится в PG (`refresh_tokens` – уже есть миграция).
- При logout: access-token добавляется в Redis blacklist с TTL = оставшийся срок.
- При logout-everywhere: все refresh-токены пользователя удаляются из PG + все access-токены в blacklist.

### 8.3 Транспорт

- Для альфы локально: http без TLS (друзья коннектятся через cloudflared-туннель, TLS терминируется на cloudflared).
- При переезде на Hetzner: nginx + Let's Encrypt.

### 8.6 Распространение альфы

Друзья-тестеры не компилируют Tauri сами – мы им отдаём готовый установщик:

- `npm run tauri build` на Windows 10/11 собирает `.msi` (и/или `.exe`-инсталлятор).
- Сборка подписывается (для альфы можно без подписи, Windows SmartScreen покажет предупреждение – объясним тестерам).
- Релизы хостим приватно (например, Google Drive / Dropbox / прямая ссылка с нашего сервера с токеном).
- URL сервера (cloudflared-туннель) вшит в сборку на момент билда через `VITE_API_BASE_URL`. При смене туннеля – пересобираем и раздаём новую версию.
- Tauri auto-updater – вне альфы, но предусмотреть схему URL-ов так, чтобы потом легко добавить.

### 8.4 CSRF

- Tauri-приложение не в браузере, CSRF не применим (нет cookies в том же origin как website).
- JWT в Authorization header.

### 8.5 Что ещё отложено

- 2FA (есть упоминание в скиллах, но в альфе не делаем).
- Email verification (register → email confirm). В альфе: регистрация сразу активна, без email-подтверждения. **Риск:** мусорные регистрации. Для закрытой альфы с ссылкой-приглашением – приемлемо.

## 12. Критерии готовности альфы (acceptance criteria)

Альфа считается готовой, когда **все** пункты ниже проходят с живыми пользователями:

1. ✅ Два разных пользователя регистрируются (email+пароль) на двух разных машинах.
2. ✅ Оба видят главный экран по Figma.
3. ✅ Оба открывают Категории → Онлайн карту мира → видят 3D-мир.
4. ✅ Оба видят минимум 10 камер в 3D-мире (thumbnail на маркерах).
5. ✅ Пользователь A кликает на маркер Токио → видит живой видеопоток в 3D-плоскости.
6. ✅ A жмёт "Смотреть вместе" → создаётся комната, копируется ссылка.
7. ✅ A отправляет ссылку B через внешний мессенджер.
8. ✅ B кликает на ссылку → в Placebo открывается та же комната → видит ту же камеру.
9. ✅ A переключает камеру → у B камера меняется в течение 2 секунд.
10. ✅ Оба пишут в чат, сообщения видны друг у друга в течение 1 секунды.
11. ✅ Счётчик зрителей на камере показывает 2 у обоих.
12. ✅ B добавляет A в друзья по никнейму → A видит B в списке друзей.
13. ✅ Оба могут переключаться между табами (открыть второй таб с камерой в Хельсинки, первый таб с комнатой в Токио продолжает работать).
14. ✅ `< > ↻` работают внутри таба.
15. ✅ Переключатель темы светлая/тёмная работает, сохраняется после перезапуска.
16. ✅ Оба могут разлогиниться → logout работает на сервере, повторное открытие приложения требует логин.
17. ✅ Alpha-сборка `npm run tauri build` собирает `.msi` инсталлятор. Инсталлятор ставится на чистой Windows 10/11 без предустановленных Node/Rust/Docker, приложение запускается и подключается к серверу через вшитый URL.
18. ✅ Нет unhandled промисов, белых экранов, или потерянных сессий в главном сценарии.

## 13. Риски и открытые вопросы

### 13.1 Технические риски

- **Мульти-таб + 3D-виртуализация** – самый сложный инженерный кусок альфы. Риск: разломать 3D-мир при попытке виртуализации. **Митигация:** начать с простой реализации (размонтирование при уходе с таба), если это приемлемо по UX – оставить; усложнять только если нужно.
- **yt-dlp в production** – хрупко. **Митигация:** fallback на публичные HLS (Skyline, EarthCam) для критичных камер.
- **WebSocket reconnect в Tauri** – Tauri WebView может терять WS при засыпании ОС. **Митигация:** реализовать exponential backoff + resume семантику с первого дня.
- **Rust → TypeScript типогенерация** – ts-rs требует ручного триггера. **Митигация:** добавить в pre-commit hook или `npm run dev` автоматически вызывать.

### 13.2 Продуктовые риски

- **10–15 камер – хватит ли для "вау"-эффекта?** Продукт выигрывает разнообразием. **Митигация:** в альфе сделать **поиск по камерам работающим** или группировку по городам.
- **3D-мир может быть тяжёл для слабых машин тестеров.** **Митигация:** иметь "2D fallback" – простая карточка камеры без 3D. На альфе не делаем, но следим за фидбеком.

### 13.3 Решения по открытым вопросам (зафиксированы 2026-05-14)

1. **Никнейм и юзернейм – обе сущности.**
   - `display_name` (никнейм) – "Имя" + опционально "Фамилия", допускает кириллицу, пробелы, эмодзи. Уникальность **не требуется**. Показывается в UI везде, где представляется пользователь.
   - `username` – латиница + цифры + подчёркивание, регистр игнорируется при сравнении. Уникален в системе. Используется в URL профиля (`/profile/:username`), меншенах (`@username`), при добавлении в друзья.
   - Оба поля обязательны для уникальной идентификации, но в форме регистрации Figma показывает display_name как обязательное, username как опциональное. При отсутствии username на регистрации – генерируем из `display_name` + суффикс (например, транслитерация + случайный суффикс из 4 цифр), пользователь может потом изменить в Настройках.

2. **Уникальность username проверяется при submit.** Real-time validation не делаем (лишняя нагрузка на API, усложняет форму). Если username занят – возвращаем понятную ошибку и предлагаем 3 свободных варианта.

3. **Дата рождения – опциональное поле профиля.** В альфе сохраняем в БД, но не используем в продуктовой логике. Будущие применения (ориентировочно): рекомендации контента по возрасту, возрастные ограничения (18+), поздравления с днём рождения в соцсети. Поле в профиле можно скрыть от других пользователей (toggle из Figma "Скрыть в профиле дату рождения").

4. **Фото профиля – опционально, хранится в Cloudflare R2.**
   - На этапе регистрации можно пропустить – аватар по умолчанию генерируется из первой буквы display_name на цветном круге (цвет выбирается детерминированно из user_id).
   - Upload идёт двухшагово: клиент запрашивает presigned URL у нашего API, заливает прямо в R2, возвращает API ключ файла – API сохраняет ссылку в users-таблице.
   - В альфе: если R2 ещё не настроен, fallback – сохранять локально на сервере в `./uploads/avatars/{user_id}.jpg` и отдавать через `/uploads/avatars/{user_id}.jpg`. Это не идеал, но работает для 10–50 тестеров. Миграция на R2 – первая задача после альфы.

5. **Invite-ссылка – короткий код.**
   - Формат: `https://<host>/r/<code>`, где `code` – 8 символов base58 (без похожих: `0`, `O`, `I`, `l`).
   - Вероятность коллизии на 8 символах base58 ≈ 1 к 2.2 × 10¹⁴, более чем достаточно.
   - Код разрешается в `room_id` через endpoint `GET /api/v1/rooms/by-invite/:code`.
   - Срок жизни invite-кода: 24 часа с момента создания (для альфы – можно сделать бессрочный, если 24h мешает сценарию демо).
   - Один код привязан к одной комнате, но многоразовый (несколько друзей могут перейти по одной ссылке).

6. **Поиск друзей – по обоим полям, как в TikTok.**
   - Запрос: `GET /api/v1/users/search?q=...` с полнотекстовым поиском по `username` и `display_name`.
   - Реализация на PostgreSQL: **GIN-индекс + `pg_trgm` extension** (trigram similarity) на оба поля. Это даёт fuzzy search: опечатки, частичные совпадения, case-insensitive.
   - Ранжирование: точное совпадение username > начало username > точное совпадение display_name > trigram similarity.
   - Пагинация: 20 результатов на страницу.
   - В альфе – базовый поиск, оптимизацию (кэш популярных запросов в Redis, Elasticsearch) добавляем в бета.
